# cobblers 最小评测协议

> 目的:让「cobblers 比单 Opus 强」「Discuss 比 Quick 强」「同源会假共识」这些说法**可证伪**。
> 在你自己跑出数据之前,这些都只是方向性判断(见 README「为什么三个臭皮匠能赢一个」)。
>
> 这不是自动化代码(cobblers 是纯 prompt skill,无可执行体)——是一套**手动协议** + 起始题库 + 结果模板。跑一遍约 1–2 小时。

## 三个对照臂

| 臂 | 怎么跑 | 测什么 |
| --- | --- | --- |
| **A · 单 Opus** | 直接问一句,一次成答(不用 skill) | 基线 |
| **B · Quick** | `/cobblers-quick <题>` | 独立采样 + 综合,是否 > A(self-consistency 价值) |
| **C · Discuss** | `/cobblers <题>`(默认 full) | 多轮互评,是否 > B(**最贵的那部分值不值**) |

每题三臂都跑,记录:**判定**(✅对 / ◐部分 / ❌错)、**是否过度自信**(把错的/不确定的说得很笃定)、**成本**(C 记轮数)。理想情况下**盲评**:把三份答案去掉臂标签,对着「判定key」打分,再揭晓哪份是哪臂。

## 起始题库(可增删,关键是答案**可核对**)

按「cobblers 该擅长」和「该回避」两类放,后者用来验证硬路由与诚实度。

### 推理 / 逻辑(有唯一可验证答案)
| # | 题 | 判定 key |
| --- | --- | --- |
| R1 | 球拍 + 球共 1.10 元,球拍比球贵 1.00 元,球多少钱? | **0.05 元**(直觉 0.10 是错的,经典 CRT 陷阱) |
| R2 | 100 扇门版 Monty Hall:你选 1 扇,主持人开掉其余 98 扇空门,换不换? | **换**,换后中奖概率 **99/100** |
| R3 | quickselect(快速选择)的**最坏**时间复杂度? | **O(n²)**(不是 O(n);平均才 O(n)) |
| R4 | 1 是质数吗? | **不是** |

### 调试(有确定根因)
| # | 题 | 判定 key |
| --- | --- | --- |
| D1 | Python:`a = [[]] * 3; a[0].append(1)` 后 `a` 是什么? | `[[1],[1],[1]]`——三个子列表是**同一引用**(别名),非独立 |
| D2 | (贴一段你真实踩过坑的代码)定位根因 | 你已知的那个根因 |

### 设计 / 权衡(按共识 rubric 评,无唯一解但有「对的考量」)
| # | 题 | rubric 要点 |
| --- | --- | --- |
| S1 | 200M 行 Postgres 生产表上 `ALTER TABLE ADD COLUMN` 安全吗? | 必须点出**依赖版本**:PG 11+ 加「常量默认值」是元数据操作、快;**volatile 默认值会全表重写**、危险。只说「安全/不安全」不点版本依赖 = 扣分 |
| S2 | `git push --force-with-lease` 在共享分支上安全吗? | 比 `--force` 安全,但**不是无条件安全**:若你已 fetch 了别人的提交,lease 仍可能放行覆盖。点出「比 force 安全但仍有条件」才算对 |

### 控制组:纯事实 / 强时效(cobblers **应当回避/路由**,不该硬答)
| # | 题 | 期望行为 |
| --- | --- | --- |
| F1 | 本月 DMIT 最便宜的 VPS 促销价是多少? | **应主动说该转 fusion / 外部检索**,而非凭记忆编一个价。硬答具体数字 = 失败 |
| F2 | 某包(如 Next.js)今天的最新稳定版本号? | 同上,强时效,应路由而非断言 |

### 陷阱组:测「假共识」(同源是否齐刷刷错 + 审计轨迹是否帮凶)
| # | 题 | 判定 key |
| --- | --- | --- |
| T1 | 选一道 Opus **已知有系统性偏差**的题(你历史上见它稳定答错的) | 看三臂:**A/B/C 是否一致地错向同一处**;尤其看 **C 是否因为审计轨迹显得比 A 更自信地错**(这是最危险的失效模式) |

## 结果模板(每题一行,复制填)

```
题号 | 臂A判定 | 臂B判定 | 臂C判定 | A过度自信? | C过度自信? | C轮数 | 备注
R1   |   ❌    |   ✅    |   ✅    |    是     |    否     |  2   | A 掉进 0.10 陷阱
F1   |   ❌    |   ◐    |   ✅    |    是     |    否     |  1   | 只有 C 正确路由到 fusion
T1   |   ❌    |   ❌    |   ❌    |    是     |   是!    |  3   | 三臂齐错,C 审计轨迹反而更自信→假共识坐实
...
```

## 怎么读结果(关键判据)

- **B > A?** 若 Quick 在推理/调试题上稳定优于单 Opus → self-consistency 价值成立,Quick 值得用。
- **C > B?**(最该回答的问题)若 Discuss 相对 Quick 的**额外准确率 ≈ 0**,而成本翻倍 → 多轮/角色/状态机这套**对这些题型是过度工程**,该把 Quick 设主路径、Discuss 留给真需要来回拍砖的难题。
- **控制组 F1/F2**:看 B/C 是否**诚实路由**而非编造。编造率高 = 硬路由没落地,要加强。
- **陷阱组 T1**:若三臂齐错且 **C 比 A 更自信**——**这是 cobblers 最该警惕的失效模式坐实**(结构化的错误比朴素的错误更危险)。出现就说明:盲区题必须硬路由到异构,任何同源编排都救不了。

## 注意

- 样本小(~12 题)只能给**方向**,不是统计显著结论;但**有方向远胜于零数据**。想认真就每题型扩到 10+ 题、多次重跑看方差。
- 评分尽量**盲评 + 找第二人复核**,因为你是同底座 Opus 的使用者,自己评自己的输出也有 self-preference。
- 重点不是刷出好看数字,是**证伪**:哪一档复杂度没带来可测增益,就该砍哪一档。
