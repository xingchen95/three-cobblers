# Discuss 模式 —— 3 个臭皮匠多轮讨论

主窗口 Opus（诸葛亮）是唯一的中转枢纽 + 最终整合者。三个臭皮匠子代理**之间不能直接通信**：所谓「讨论」= 主窗口每轮收集三家答案 → 用 `SendMessage` 把另两家的信息分发回每个子代理 → 各自修正 → 再收集。主窗口因此是一个**中转瓶颈**——「上下文传递模式」控制这个瓶颈漏多少信息过去。

## 上下文传递模式（可设置）

| 模式 | 互评轮转发什么 | 上下文完整度 | token | 默认 |
| --- | --- | --- | --- | --- |
| `digest` | 各家自产的**「自述卡」**（核心主张+证据+假设+待验证） | 中高（作者本人提炼，非主窗口二手摘要） | 适中 | ✅ |
| `full` | 各家**完整的 Round 1 答案全文** | 高（连论证链、证据、研究过程都给） | 高（每轮每人多吞两份长答案） | |

**怎么选**：用户用 `/cobblers --full` 或说「完整上下文 / full」→ `full`；否则 `digest`。把标志从问题文本里剥掉再传给臭皮匠。`digest` 转发的是臭皮匠**自己写的自述卡**而非主窗口二手摘要，避免「摘要失真 → 批判者打稻草人」。

## Round 1 — blind 独立发散

把用户任务**逐字原文**（不要预先消化/总结）+ 简短指令（用 web/bash 自行研究、作为多位独立专家之一、不会看到他人回答、给完整自洽的中文答案）构造成**统一 prompt**，三份完全相同，**不分配角色**。

要求每个臭皮匠在答案结尾附一张结构化「自述卡」：

```
【自述卡】
- 核心主张：3–6 条，每条一句话
- 关键证据/来源：支撑上述主张的证据，标出处（URL/命令/文献）；无一手来源的主张必须标「未验证」
- 关键假设：本答案依赖、但未必成立的前提
- 待验证/不确定：还需查证或实验才能定的点
```

**在同一条消息里 spawn 3 个 Opus 子代理**（`Agent`，`subagent_type: general-purpose`，`run_in_background: true`），记下三个 agentId。等三者全部返回。

## Round 2+ — 互评（主窗口中转 + 角色 + 议程）

三家上一轮到齐后，对**每个** agentId 用 `SendMessage` 发送（三个 `SendMessage` 放同一条消息并发）：

1. **另外两家**的信息（标 `[臭皮匠乙]` / `[臭皮匠丙]`，不发它自己的）——`digest` 转对方自述卡、`full` 转对方完整答案；
2. 当前**工作黑板**（见下）；
3. 它的**角色**（严谨者 / 发散者 / 批判者，定义见 `references/roles.md`）；
4. **互评议程**——本轮不要复述上轮结论，只做三件事：
   - ① 指出另两家方案的**隐含假设、边界条件、可能反例**；
   - ② 把每个争议点**转成一个可验证的最小动作**（实验 / 仿真 / 计算 / 检索）；
   - ③ 明确**哪些结论已经足够稳（可进共识）、哪些还只是候选方向**；
   - 并更新自己的「自述卡」。

等三者返回。

## 工作黑板（主窗口每轮维护）

每轮结束后，主窗口更新一块黑板，下一轮随互评指令下发，也是收敛判断的依据和最终整合的骨架：

```
- 已确认事实：三家独立认可、经互评仍站得住的点
- 主要分歧：谁 vs 谁、争什么、是「证据可解」还是「需用户定夺」
- 开放问题：尚未解决的子问题
- 证据缺口：哪些主张还缺一手来源
```

## 收敛判断（主窗口当调度器，每轮结束后）

**只有同时满足以下「收敛门槛」才允许进入整合**，否则再来一轮：
- 已形成明确的候选方向；
- 关键分歧已列清（要么已裁定，要么已标「需用户/实验定夺」）；
- 证据缺口已标出；
- 每个主要结论都有下一步验证路径。

三个**提前终止**条件（满足任一即停）：
- **stalled**：某轮三人都「无实质新增」→ 停，别硬凑轮数；
- **轮数硬上限**：默认 2 轮（首轮 + 1 互评），最多 3 轮；
- **ASK_USER 暂停**：若核心分歧是**根本方向之争、且要靠用户的偏好/约束/优先级才能定**（不是再多证据能解决的）→ **不要硬整合**。用 `AskUserQuestion` 把分歧做成 2–4 个选项暂停问用户，拿到回复后再继续一轮或直接整合。

## 整合（主窗口 Opus，最终交付）

**不强行压成共识，保留分歧。** 给每个关键主张标状态：✅共识 / ⚔️争议 / ❌已反驳 / 🔬待验证。输出：
- **共识**：独立提出且经互评仍站得住的点（最高置信）；
- **矛盾**：互评中正面交锋、未消解的分歧（标各方立场，能裁就裁，裁不了说清还差什么）；
- **部分覆盖**：只有部分人深入的子问题；
- **独特洞见**：仅一人提出且有价值的点；
- **盲点**：三人共同忽略或共享的错误假设（诸葛亮可自补一条）；
- **观点演化**：关键观点谁首轮提出 → 被谁用什么挑战 → 最终存活/修正/放弃（带状态标记）；
- **最终答案**：grounded 在以上——先 high-confidence 共识，纳入独特洞见，标注仍不确定处。

呈现时注明本次上下文模式（digest / full）与轮数。

## 容错与权重

- 某子代理**失败/超时** → 视为**该轮缺席**，用其余两家继续，整理时显式标注「臭皮匠 X 第 N 轮缺席」，**绝不当作沉默同意**。
- Round 1 不足 2 家返回 → 降级为可用家数的一次性整合，并说明。
- 证据 > 断言：跑过代码或读过一手来源的，权重高于凭记忆推理的，与谁无关。藏起真冲突比没讨论更糟。
