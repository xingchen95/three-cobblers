# Discuss 模式 —— 3 个臭皮匠多轮讨论（mindforge 编排增强版）

主窗口 Opus（诸葛亮）是唯一的中转枢纽 + **调度器** + 最终整合者。

**关键约束（决定整个实现）**：当前 Claude Code **无法续接一个已 spawn 的子代理**——没有 `SendMessage` 之类的工具，`Agent` 也没有续接参数。子代理一旦返回就结束。所以「多轮讨论」**不是**靠续接同一个子代理，而是靠**每轮重新 spawn 全新子代理**，由主窗口把上一轮的上下文**注入新 prompt** 来重建。主窗口负责保存每轮每家的答案、证据卡和研究状态，供下一轮注入。

> 本模式移植了 mindforge 的五项编排机制：①首轮方法论分叉 ②结构化证据卡 ③5-角色池（见 `roles.md`）④研究状态机 ⑤动态调度器（含 REFOCUS）。但保持 cobblers 的纯 Opus、零依赖、主窗口 re-spawn 形态——无持久层、无异构、无外部代码。

## 上下文传递模式（可设置）

控制互评轮注入给每个臭皮匠的「另两家信息」有多少：

| 模式 | 注入另两家的 | 完整度 | token | 默认 |
| --- | --- | --- | --- | --- |
| `digest` | 各家自产的**「自述卡」**（含证据卡） | 中高（作者本人提炼） | 适中 | ✅ |
| `full` | 各家**完整答案全文** | 高（连论证链都给） | 高 | |

**怎么选**：用户用 `/cobblers --full` 或说「完整上下文」→ `full`；否则 `digest`。把标志从问题文本剥掉再用。**权衡**：互评的首要价值是挑对方的隐含假设，而隐含假设往往就藏在被 digest 压掉的论证链里——所以**高风险 / 调试 / 找反例类问题建议主动 `--full`**；digest 更适合省 token 的低风险场景。

## Round 1 — blind 独立发散 + 轻量方法论分叉

把用户任务**逐字原文** + 简短指令构造成统一 prompt，三份**主体完全相同、不分配角色**。**唯一差异是一行「思考侧重」**——主窗口先一句话判题型，从正交方法论三元组里给甲/乙/丙各一个（只分叉「怎么推」，不分叉「推向哪」，保住「独立得出共识」的信号价值）：

| 题型 | 方法论三元组（甲 ｜ 乙 ｜ 丙） |
| --- | --- |
| 理论 / 物理 / 数学 | 第一性原理 ｜ 数值/极限仿真 ｜ 边界与反例 |
| 工程 / 设计 / 策略 | 第一性原理 ｜ 跨域类比 ｜ 红蓝队压测 |
| 调试 / 故障 | 逆向工程 ｜ 数据驱动归纳 ｜ 最小复现 |
| 拿不准 | 第一性原理 ｜ 类比迁移 ｜ 逆向工程 |

注入话术（仅这一行三家不同）：「**本轮思考侧重：【X】**——<一句话定义>。这是侧重不是枷锁，若不适配本题就如实说明并灵活调整。」

通用指令（三家相同）：用 web/bash 自行研究、作为多位独立专家之一、不会看到他人回答、给完整自洽中文答案；**若 web/bash 工具不可用或返回异常，必须在自述卡显式标「未联网核实」，不得假装查过**。

每家结尾附「自述卡」，证据部分用**结构化证据卡**（移植 mindforge `EvidenceCard`）：

```
【自述卡】
- 核心主张：3–6 条，每条一句话
- 证据卡（每条关键证据一张）：
    · 支持/反驳：本证据支持或反驳哪条核心主张（编号）
    · 来源：URL / 命令 / 文献（一手）；无一手来源标「未验证·记忆推理」
    · 可信度：高（一手实验 / .gov / .edu / 官方规范）｜ 中（厂商文档 / 二手综述）｜ 低（博客 / 论坛 / 纯记忆）
- 关键假设
- 待验证 / 不确定
- 方法论增益：我这个思考侧重让我看到 / 排除了什么别人可能漏的？（答不出 = 本次分叉空转，如实标注）
```

**同一条消息 spawn 3 个 Opus 子代理**（`Agent`，`subagent_type: general-purpose`，`run_in_background: true`）。等三者全部返回，**主窗口保存三份答案 + 三张自述卡**，并把证据卡抽进下面的「研究状态」。

## Round 2+ — 互评（先选角 → re-spawn → 注入上下文）

**先选角**：主窗口依题型从 5 角色池（见 `roles.md` 选角表）选 3 个，按 spawn 顺序映射甲 / 乙 / 丙（拿不准就用默认：严谨者 / 发散者 / 批判者）。一场讨论内角色固定。

**不续接旧子代理。为每个臭皮匠 re-spawn 一个新 Opus 子代理**（同一条消息并发），prompt 由主窗口拼装：

```
你是臭皮匠【甲】，正在参加一场多专家讨论的第 N 轮。

你上一轮的立场（完整答案）：
<主窗口注入：甲上一轮的答案>

另外两位专家的观点：
[乙] <按模式注入：乙的自述卡 或 全文>
[丙] <按模式注入：丙的自述卡 或 全文>

当前研究状态：
<主张表 / 分歧 / 证据缺口 / 验证任务 / 已拒方案 / 开放问题>

你的角色：<从 roles.md 选定的角色，含四段职责>

互评议程（本轮不复述上轮，只做三件事）：
① 指出乙丙的隐含假设、边界条件、可能反例；
② 把每个争议点转成一个可验证的最小动作（实验/仿真/计算/检索）；
③ 明确哪些结论已足够稳（可进共识）、哪些仍是候选方向。
基于以上更新你的立场：保留/修正/放弃哪些观点 + 为什么，并给出更新后的「自述卡」（含证据卡）。
```

等三者返回，主窗口更新研究状态。

## 研究状态（主窗口每轮维护，移植 mindforge ResearchState）

把原「工作黑板」升级为结构化研究状态：

```
研究状态：
- 主张表 claims：每条关键主张 ｜ 状态 ✅确认 / ⚔️争议 / ❌已反驳 / 🔬待验证 ｜ 谁提出 ｜ 支撑证据卡（带可信度）
- 主要分歧（≤3，结构化）：争议点一句话 ｜ 各方立场（甲/乙/丙）｜ 证据（谁有一手来源、谁更站得住）｜ 下一步验证；无实质分歧则记「本轮基本一致」
- 证据缺口 evidenceGaps：哪些主张还缺一手来源
- 验证任务 validationTasks：待执行 / 已执行的最小验证动作（谁做、结果如何）
- 已拒方案 rejectedIdeas：被否决的方向 + 为什么（防止后续轮重复讨论已死的方向）
- 开放问题 openQuestions：尚未解决的子问题
```

## 动态调度器（主窗口当调度器，每轮互评后判断下一步）

移植 mindforge 的 `pickNextSpeaker`。每轮结束后，主窗口在五种动作里选一个：

1. **CONTINUE（再来一轮）**：分歧未决但仍在收敛。**不必每轮 re-spawn 全部 3 个**——只 re-spawn 与未决分歧最相关的角色（分歧在证据 → 证据者 + 批判者；分歧在方向 → 发散者 + 严谨者），省 token、更聚焦；但**每轮至少 2 个**以保住交叉视角，且不让批判职能缺席。
2. **CONVERGED（收敛 → 整合）**：满足下面「收敛门槛」→ 进整合。
3. **REFOCUS（纠偏）**：讨论偏离根问题（钻枝节 / 跑题 / 各说各话）→ 下一轮 prompt 注入「回到根问题 X，停止纠缠枝节 Y」。**同一场 REFOCUS 累计 2 次仍偏 → 自动升级为 ASK_USER**（移植 mindforge `refocusCount >= 2`）。
4. **ASK_USER（问用户）**：核心分歧是**根本方向之争、要靠用户偏好/约束才能定**（非证据可解）→ 用 `AskUserQuestion` 把分歧做成选项暂停问用户（**问题简短 <20 字，2–4 个具体、互不重叠的方向选项**），拿到回复再继续或整合。
5. **STALLED（停）**：某轮三人都「无实质新增」→ 停，直接进整合。

**收敛门槛**（CONVERGED 的前置）：
- **格式条件**：已形成明确候选方向；关键分歧已列清（已裁定或已标「需用户/实验定夺」）；证据缺口已标出；每个主要结论有下一步验证路径。
- **实质条件（防「格式齐全的假收敛」）**：上面只是「格式齐全」——三个同源 Opus 完全可能一致地、自信地错，同时把分歧表/缺口/验证路径填得工整，产出「包装精美的错误共识」。所以加硬门槛：**至少一条关键主张要有一手来源支持（真的跑了代码 / 查了文献 / 做了计算），而非纯记忆推理**；若三家收敛却全部主张都只有记忆级证据、无任何真实执行过的验证（`validationTasks` 全空），**不算收敛**，强制 CONTINUE 一轮并要求产出 ≥1 个真实执行的验证。

**轮数硬上限**：默认 2 轮（首轮 + 1 互评），最多 3 轮。

## 整合（主窗口 Opus，最终交付）

**先校准你自己的裁判偏差**：你（裁判）与三个臭皮匠是同一底座 Opus，会本能偏好**表述/推理方式像你自己**的那份（self-preference），这会抵消集成本该带来的去偏。所以排序以**证据强度**（证据卡可信度、是否跑过代码 / 有一手来源）而非表述流畅度为准，并对你**最认同**的那份额外做一次主动证伪；三家一致**不自动等于正确**（见上「收敛门槛」实质条件）。

**不强行压成共识，保留分歧。** 沿用研究状态里的 claims 状态（✅共识 / ⚔️争议 / ❌已反驳 / 🔬待验证）。输出：
- **共识** / **矛盾** / **部分覆盖** / **独特洞见** / **盲点**（诸葛亮可自补一条）；
- **观点演化**：关键观点谁首轮提出 → 被谁用什么挑战 → 最终存活/修正/放弃（带状态）；并把 `rejectedIdeas`（被否决的方向）与 `validationTasks`（执行过的验证及结果）纳入审计轨迹；
- **最终答案**：grounded 在以上，先 high-confidence 共识，纳入独特洞见，标注仍不确定处。

呈现时注明上下文模式（digest / full）、轮数、本场选用的三角色。

## 容错与权重

- 某子代理**失败/超时** → 视为**该轮缺席**，用其余两家继续并标注，**绝不当作沉默同意**。家数从 3 降到 2 时去相关性进一步变弱，**共识置信相应下调一档**；若缺席的恰是「批判者」，主窗口须临时补上反驳职能，别让这一轮没人专职挑错。
- Round 1 不足 2 家返回 → 降级为可用家数的一次性整合，并说明。
- 证据 > 断言：跑过代码/读过一手来源的（证据卡可信度高的），权重高于凭记忆推理的。藏起真冲突比没讨论更糟。
