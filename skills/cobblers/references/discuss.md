# Discuss 模式 —— 3 个臭皮匠多轮讨论

主窗口 Opus（诸葛亮）是唯一的中转枢纽 + 最终整合者。三个臭皮匠子代理**之间不能直接通信**：所谓「讨论」= 主窗口每轮收集三家答案 → 用 `SendMessage` 把另两家的观点分发回每个子代理 → 各自修正 → 再收集。

## Round 1 — blind 独立发散

把用户任务**逐字原文**（不要预先消化/总结）+ 简短指令（用 web/bash 自行研究、作为多位独立专家之一、不会看到他人回答、给完整自洽的中文答案）构造成**统一 prompt**，三份完全相同，**不分配角色**。

**在同一条消息里 spawn 3 个 Opus 子代理**（`Agent`，`subagent_type: general-purpose`，`run_in_background: true`），记下三个 agentId。等三者全部返回。

## Round 2 — 互评（主窗口中转 + 加角色）

三家 Round 1 答案到齐后，对**每个** agentId 用 `SendMessage` 发送（三个 `SendMessage` 放同一条消息并发）：
1. **另外两家**的 Round 1 核心观点（标 `[臭皮匠乙]` / `[臭皮匠丙]`，不要把它自己的答案再发回去）；
2. 给它指派的**角色**（严谨者 / 发散者 / 批判者，定义见 `references/roles.md`）；
3. **互评指令**：明确 保留/修正/放弃 哪个观点 + 为什么、关键假设与最强反例、证据缺口、一个最低成本验证；不要泛泛重答。

等三者返回修正稿。

## 收敛判断（主窗口当调度器）

读三份修正稿，自行判断（不需额外模型）：
- 仍有**重大未决分歧**且轮数 < 上限 → 发起 Round 3：再 `SendMessage` 互评，**只聚焦那几个未决分歧点**；
- 否则 → 停，进入整合。
- **轮数硬上限默认 2 轮（首轮 + 1 互评），最多 3 轮。** 同模型多轮易趋同，边际收益递减，不要无限讨论。

## 整合（主窗口 Opus，最终交付）

**不强行压成共识，保留分歧。** 输出：
- **共识**：独立提出且经互评仍站得住的点（最高置信）；
- **矛盾**：互评中正面交锋、未消解的分歧（标各方立场，能裁就裁，裁不了就说清还差什么）；
- **部分覆盖**：只有部分人深入的子问题；
- **独特洞见**：仅一人提出且有价值的点；
- **盲点**：三人共同忽略或共享的错误假设（诸葛亮可自行补一条）；
- **观点演化**（Discuss 专属）：关键观点谁首轮提出 → 互评中被谁用什么挑战 → 最终存活/修正/放弃。这是讨论模式相对 Quick 模式最大的增量价值；
- **最终答案**：grounded 在以上分析——先 high-confidence 共识，纳入独特洞见，标注仍不确定处。

## 容错

- 某子代理**失败/超时** → 视为**该轮缺席**，用其余两家继续，整理时显式标注「臭皮匠 X 第 N 轮缺席」，**绝不当作沉默同意**。
- Round 1 不足 2 家返回 → 本次降级为可用家数的一次性整合，并说明。

## 与权重

证据 > 断言：跑过代码或读过一手来源的臭皮匠，权重高于凭记忆推理的，与谁无关。诚实对待置信度与分歧——藏起真冲突比没讨论更糟。
