# 三个臭皮匠（Cobblers）

核心机制是 **独立，再综合**。同一个难题发给 3 个 Opus 子代理，每个都带 web 搜索和 bash、互相看不到对方的工作；然后主窗口 Opus 把三份答案碰撞、整合。多样性靠**独立性**收割，不靠人为分配角色——首轮三人拿到一字不差的同一个任务。

**一条硬规则：主窗口 Opus 永远是「诸葛亮」（裁判 + 整合者），流水线不可逆。** 臭皮匠子代理无法回头 spawn 主窗口，所以整合永远在主窗口发生，且它读所有答案时是旁观者，不偏袒自己写的那份。

## 选模式

| 模式 | 命令 | 流程 | 何时用 |
| --- | --- | --- | --- |
| **Discuss（主打）** | `/cobblers` | 首轮 blind 独立 → 多轮互评（加角色）→ 整合 | 真正需要来回碰撞、互相挑错的难题 |
| **Quick** | `/cobblers-quick` | 3 个 Opus 一次性 blind → 直接整合 | 想要多视角但不必多轮，快且省 |

默认路由：用户提到「讨论 / 辩论 / 多轮 / 互相」→ **Discuss**；只想「多个意见 / cross-check / 几个模型看看」→ **Quick**；拿不准 → **Discuss**。用户用了 `/cobblers` 或 `/cobblers-quick` 就按命令来。

## Discuss 模式

照 `references/discuss.md` 执行（首轮独立 → re-spawn 注入互评 → 收敛判断 → 整合）。三个角色的定义见 `references/roles.md`。

**上下文传递模式（可设置）**：互评轮另两家的信息怎么给——`digest`（默认，转发各家自产的「自述卡」：核心主张+证据+假设，省 token、是作者本人提炼不是主窗口二手摘要）或 `full`（转发各家完整 Round 1 全文，连论证链都给，最完整、token 更高）。用户用 `/cobblers --full`、或说「完整上下文 / full」→ `full`；否则 `digest`。详见 `references/discuss.md`。

## Quick 模式

1. **同一条消息** spawn 3 个 Opus 子代理（`Agent` 工具，`subagent_type: general-purpose`，`run_in_background: true`）。每个 prompt = 用户任务**逐字原文** + 简短指令：用 web/bash 自行研究、作为多位独立专家之一、不会看到他人回答、给一份完整自洽的中文答案。三份 prompt 完全相同，不分配角色。
2. 等三者全部返回。任一失败/超时 → 视为**缺席**，用其余继续，**绝不当作沉默同意**；不足 2 份则说明并降级。
3. 主窗口整合（不做平均、不抹平冲突）：**共识 / 矛盾 / 部分覆盖 / 独特洞见 / 盲点 / 最终答案**。独立同意 = 最高置信；跑过代码或读过一手来源的，权重高于凭记忆推理的。

## 通用规则

- 子代理之间**不能直接通信**——任何「互看」必须由主窗口中转。
- **首轮永远 blind**：构造 prompt 时不要预先消化/总结任务，原样传递；不要分配角色。
- 最终答案默认用**中文**（除非用户要求其他语言）。
- **前提**：会话模型为 Opus 4.8；子代理与裁判都继承会话模型。换成别的模型，"Opus" 只是名义。
- **呈现**：先给最终答案，再给审计轨迹（五段分析；Discuss 多一节「观点演化」）。注明跑的是 Discuss 还是 Quick、几个臭皮匠参与、有无缺席。
- **成本**：Quick ≈ 3× 单次；Discuss ≈ 3–5×。低风险/简单问题直接单模型答更划算——别为了用而用。
