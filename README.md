# 三个臭皮匠 · Three Cobblers

**三个臭皮匠，顶个诸葛亮。** 把一道难题交给 3 个 Opus「臭皮匠」**各自独立**琢磨、再**互相拍砖**，最后由主窗口 Opus「诸葛亮」整合成一个高于任何单个模型的答案。

这是一个 [Claude Code](https://claude.com/claude-code) skill。它的机制是 **独立，再综合**：同一个问题同时发给三个 Opus 子代理，每个都带 web 搜索和 bash、互相看不到对方的回答；然后主窗口把三份答案碰撞、整合。让一个模型「臭皮匠化」成三个独立视角的关键，是**独立性**——哪怕同一个 Opus 跑三遍，推理路径、工具调用、引用来源也会发散，把这种发散收割再综合，就胜过让它单跑一次。所以这里**没有人为分配的「角色」去污染首轮独立性**：第一轮三人拿到的是一字不差的同一个任务。

```
                          ┌─────────────┐
                     ┌──▶ │  臭皮匠 A   │ ─┐   Round 1：各自闷头想（blind，互不可见）
                     │    └─────────────┘  │
   问题 ──▶ 主窗口 ──┼──▶ │  臭皮匠 B   │ ─┼─▶ Round 2+：互相拍砖（主窗口中转）
        （诸葛亮）   │    └─────────────┘  │   严谨者 · 发散者 · 批判者
                     └──▶ │  臭皮匠 C   │ ─┘
                          └─────────────┘
                                 │
                                 ▼
                          诸葛亮整合（保留分歧，不强压共识）
                   共识 · 矛盾 · 独特洞见 · 盲点 · 观点演化 · 最终答案
```

**一条硬规则：主窗口 Opus 永远是「诸葛亮」（裁判 + 整合者），流水线不可逆。** 臭皮匠子代理没法回头 spawn 主窗口，所以整合永远发生在主窗口；而且它读三份答案时是旁观者，不会偏袒某一份自己写的。

## 两种模式

| 模式 | 命令 | 流程 | 何时用 |
| --- | --- | --- | --- |
| **Discuss（主打）** | `/cobblers` | 首轮 blind 独立 → 多轮互评（加角色）→ 整合 | 真正需要来回碰撞、互相挑错的难题 |
| **Quick** | `/cobblers-quick` | 3 个 Opus 一次性 blind → 直接整合 | 想要多视角但不必多轮，快且省 |

**Discuss** 借鉴了多 AI 讨论编排的思路：第 1 轮三人独立发散（互不可见）；第 2 轮起，主窗口把「另两家的观点」用 `SendMessage` 喂回每个臭皮匠，并分派 **严谨者 / 发散者 / 批判者** 三个角色让他们互评、反驳、修正；主窗口判断收敛（硬上限 2–3 轮），最后整合。比 Quick 多了「互相挑错」这一步，答案往往比任何单份更正确。

## 安装

```bash
git clone https://github.com/xingchen95/three-cobblers.git
cd three-cobblers
./install.sh
```

把 skill 装到 `~/.claude/skills/cobblers`、命令装到 `~/.claude/commands`。重启 Claude Code 或 `/reload-skills` 后生效。

> 用 `CLAUDE_CONFIG_DIR=/path/to/.claude ./install.sh` 可改安装位置。

## 用法

三种等价方式：

- **自然语言**（自动触发，默认走 Discuss）：
  > 用三个臭皮匠讨论一下：在 200M 行的 Postgres 生产表上 `ALTER TABLE ADD COLUMN` 安全吗？
- **命令**：
  ```
  /cobblers        我的 JWT 刷新轮换设计有重放漏洞吗？
  /cobblers-quick  git push --force-with-lease 在共享分支上真的安全吗？
  ```

每次输出统一结构：**最终答案**在最上面，下面是审计轨迹——**共识 / 矛盾 / 部分覆盖 / 独特洞见 / 盲点**，Discuss 模式再加一节 **观点演化**（某观点谁先提出 → 被谁拍砖 → 最终存活/修正/放弃），让你看清答案是怎么拼出来的。

## 前提

- **Claude Code**，且会话模型为 **Opus 4.8**——臭皮匠子代理和诸葛亮裁判都继承会话模型；换成别的模型，"Opus" 只是名义上的。
- **零外部依赖**：只用 Opus 子代理（`Agent` 工具 + `SendMessage`），不需要任何外部 CLI，开箱即用。

## 成本与延迟

- **Quick** ≈ 3× 单次回答的 token，跑得和最慢的臭皮匠一样慢。
- **Discuss** ≈ 3–5×（多轮 × 3 子代理）。

这是刻意的取舍：多花钱，换「不在代价高的地方自信地错」。低风险或简单问题，直接让一个模型答更划算——别为了用而用。

## 为什么三个臭皮匠能赢一个

集成多个独立答案再综合，能稳定胜过单个模型；而且其中相当一部分提升来自**综合这一步本身**，不只是混合不同架构——同一个模型两次独立运行、再综合，就能赢过它单跑一次。「三个臭皮匠」把这套「独立 → 综合」的流程，原地实现在 Claude Code 里。
